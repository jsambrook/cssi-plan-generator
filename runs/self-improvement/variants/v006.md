## Principles applied

### Principle #23: Feedback
**Question:** "What if the output improved the next input automatically?"

Currently each run produces `results.tsv` — a structured log of which moves were applied and what scores resulted. But the move selection heuristic (`references/select.md`) only reads the *current run's* `results.tsv`. It ignores all previous runs.

**Injection:** At the start of each new run, the agent scans all existing `runs/*/results.tsv` files to build a cross-run move effectiveness profile. This answers: "Across all ideas I've refined, which moves tend to produce the biggest score improvements, and which tend to fail?" The profile is advisory (different ideas have different needs) but gives the agent a prior instead of starting cold every time.

### Principle #25: Self-Service
**Question:** "Can the system improve itself from its own byproducts?"

The cross-session report (`runs/cross-session-report.md`) already exists as a concept, but it's triggered manually. The deeper opportunity: every run's *iteration log* contains narrative explanations of why moves worked or failed. This is exactly the kind of meta-knowledge that should accumulate.

**Injection:** The agent maintains a `runs/move-history.md` file — a living document updated at the end of each run. It contains:
- One line per move type summarizing its track record across all runs (success rate, typical score impact, when it tends to work vs. fail)
- Notable discoveries (e.g., "Drift detection was invented during the self-improvement run and should be part of the standard methodology")

This file is read at the start of every run alongside the reference files. It's the tool's memory — built from its own output, requiring no additional human effort.

## Idea

The cssi-plan-generator is an auto-improving business idea refinement tool that gets better with use.

### Learning loop (new in v006)

Each run produces structured data (`results.tsv`) and narrative explanations (iteration log). This data now feeds forward:

1. **Cross-run move profile:** At the start of each new run, the agent reads all existing `results.tsv` files and builds a move effectiveness profile. This gives move selection a prior based on actual experience rather than starting from the generic heuristics alone.

2. **Move history file (`runs/move-history.md`):** Updated at the end of each run. One line per move type with its track record. Notable methodological discoveries are captured here. Read at the start of every run.

3. **Existing mechanisms preserved:** The cross-session report remains a manual trigger for deep analysis. The move history file is the lightweight, automatic version.

## Red team (pre-scoring criticisms)

1. The cross-run move profile is only useful after multiple runs. For a new user with zero or one run, there's no data. The system degrades gracefully (falls back to generic heuristics) but the feature is invisible until run 3+.
2. `move-history.md` is agent-authored narrative about its own performance — the same self-evaluation concern from v001. The structural mitigation is that `results.tsv` is numerical and verifiable, so the narrative can be checked against the data.
3. There's a risk of overfitting: if early runs happen to succeed with Move 5, the agent may over-select Move 5 in future runs even when it's not appropriate. The advisory framing ("prior, not rule") mitigates this, but the agent's actual behavior is hard to predict.
