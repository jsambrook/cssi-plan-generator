# Variant v005

## Move applied
Move 4: Improve clarity

## Move rationale
All dimensions are clustered in the 68-74 range with Testability (70) as the bottleneck. Target 2 (first-run experience for a stranger) is unaddressed. The tool needs to be clear enough that someone cloning the repo can start with one sentence. This is a clarity problem, not a feature problem.

## Analysis: what blocks a stranger today?

1. **CLAUDE.md title says "Proposal Generator"** — stale name, creates confusion immediately.
2. **CLAUDE.md lists 13 reference files in a table** — a stranger doesn't need to know about reference files. The agent reads them internally.
3. **No explicit "just say your idea" instruction** — the README says it, but CLAUDE.md (which the agent actually reads) frames it as a multi-step process with four named questions.
4. **The agent's internal orientation is exposed** — reference file purposes, output layout details, and cross-session methodology are operational details the agent needs, not the user.
5. **README and CLAUDE.md overlap significantly** — a stranger reads the README first, then the agent reads CLAUDE.md. They serve different audiences but contain similar content.

## Proposed redesign

Split CLAUDE.md into two concerns:
- **User-facing instructions** stay in CLAUDE.md but are radically simplified. Three sections: (1) what to say to start, (2) what to say to iterate, (3) what to say to compare runs.
- **Agent-facing methodology** is already in the reference files. CLAUDE.md should point the agent to the reference files, not duplicate their content.

The README stays as-is — it's for someone browsing GitHub, not for someone inside Claude Code.

## Proposed CLAUDE.md rewrite

```markdown
# Plan Generator

An auto-improving business idea refinement tool. Tell it your idea, and it will
shape it, research it, and iterate on it to make it more concrete, valuable, and
testable.

## Quick start

**Start a new idea:** Just describe it.

> "A tool that helps plumbers schedule jobs."
> "What if BNI chapters had a shared CRM?"
> "I want to sell fractional COO services to SaaS startups."

The agent will research the space, ask a few clarifying questions, and write a
seed document for your approval.

**Run iterations:** Once you've approved the seed, say how many:

> "Run 8 iterations."

The agent iterates autonomously and writes a report when done.

**Compare runs:** When you have multiple sessions:

> "Compare all runs."

---

## For the agent

This section is read by the Claude Code agent, not the user.

### Methodology

Read the reference files in `references/` before starting any session:

- `references/mutate.md` — 11 improvement moves
- `references/score.md` — scoring rubric (five dimensions, bottleneck comparison)
- `references/select.md` — move selection heuristics
- `references/resolve.md` — TOC tension resolution (Evaporating Cloud)
- `references/format.md` — file layout and naming conventions
- `references/report.md` — report structure
- `references/toc-thinking-processes.md` — full TOC TP toolkit
- `references/cross-session.md` — cross-session analysis
- `references/evaporating-cloud.md` — EC deep dive
- `references/mafia-offer.md` — Mafia Offer methodology
- `references/goldratt-technology-rules.md` — Goldratt's technology assessment
- `references/sales-process-engineering.md` — Roff-Marsh SPE
- `references/triz.md` — TRIZ toolkit

### Seed shaping

During seed shaping, use WebSearch and WebFetch to research the problem space,
customer segment, and competitive landscape. Draft proposed answers to clarifying
questions from your research; let the user correct rather than answer from scratch.
Write the seed to `runs/<session-slug>/seed.md`. Get user confirmation before
iterating.

### Iteration loop

Run the full iteration budget autonomously. No mid-run confirmations. Apply one
move per iteration, score the variant, update current-best if it wins on the
bottleneck comparison rule. Every 3 iterations, re-read the seed and check for
drift.

### Self-evaluation integrity

- Red team: write 2-3 specific criticisms before scoring each variant
- Anchored justifications: every score references a concrete element
- Regression check: score increases >10 points require explicit justification
- Devil's advocate: required before any score of 85+
- Drift detection: every 3 iterations, compare current-best to seed
```

## Idea

The cssi-plan-generator is an auto-improving business idea refinement tool. v005 proposes a CLAUDE.md rewrite that separates user-facing instructions from agent-facing methodology.

**Key changes:**
1. **Title fixed:** "Plan Generator" instead of "Proposal Generator"

5. **Seed shaping updated:** reflects the "draft-and-correct" pattern from v003.
6. **Iteration loop updated:** reflects autonomous operation, no mid-run confirmations, drift detection.
7. **Self-evaluation integrity section added:** codifies the mechanisms from v001 and v004.

## Red team (pre-scoring criticisms)

1. The CLAUDE.md rewrite hasn't been applied yet — it's a proposal, not an implementation. Scoring should reflect what exists, not what's planned.
2. The "For the agent" label is a convention, not a guarantee. A curious user will read it. This is fine (transparency) but means the agent section still needs to be comprehensible, not just machine-readable.
3. Removing the reference file table from the user section means a power user who wants to understand the methodology must look in the agent section or the README. This is a mild friction increase for advanced users.
