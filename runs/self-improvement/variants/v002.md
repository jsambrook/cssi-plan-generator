# Variant v002

## Move applied
Move 1: Make it more concrete

## Move rationale
Testability is the bottleneck (55). The idea says "improve the tool" but doesn't specify what a successful improvement looks like. Without concrete targets, you can't tell whether an iteration actually helped.

## Idea

The cssi-plan-generator is an auto-improving business idea refinement tool. This run applies the tool to itself to produce specific, implementable improvements across three concrete targets:

### Target 1: Reduce manual steps in the methodology

**Current state:** The reference docs contain at least five places where the agent is instructed to ask the user for input or confirmation that could be automated or deferred.

**Testable outcome:** Identify each manual-action instruction in the reference docs. For each, classify as (a) genuinely requires human judgment, (b) could be automated now, or (c) could be deferred to end-of-run review. Produce a revised methodology where category (b) items are automated and category (c) items are batched.

**Success criterion:** The number of mid-run human interruptions drops by at least 50% without reducing output quality.

### Target 2: First-run experience for a stranger

**Current state:** A stranger cloning the repo must read CLAUDE.md, understand the reference file structure, and figure out how to start. There is no onboarding path.

**Testable outcome:** A new user can go from `git clone` to a running iteration loop by saying one sentence to the agent. The agent handles all orientation internally.

**Success criterion:** The README's "Usage" section is sufficient. No separate onboarding doc needed. The agent reads its own reference files without being told to.

### Target 3: Scoring integrity under pressure

**Current state:** v001 proposed four anti-inflation mechanisms. They haven't been tested.

**Testable outcome:** By the end of this 8-iteration run, the red team step, anchored justifications, and regression check have been applied to every variant. Any instance where a mechanism caught a would-be inflation is documented.

**Success criterion:** At least one score was revised downward during this run because a mechanism flagged it.

### Self-evaluation integrity

Maintained through four structural mechanisms (from v001):
- **Red team step**: Before scoring, write 2-3 specific criticisms. Recorded in the variant file.
- **Anchored justifications**: Every score references a concrete element.
- **Regression check**: Score increases >10 points require explicit justification.
- **Bottleneck comparison**: Lowest score determines the winner.

## Red team (pre-scoring criticisms)

1. Target 1 ("reduce manual steps by 50%") is measurable but the baseline hasn't been established yet — I haven't actually counted the manual-action instructions in the reference docs. The 50% target is arbitrary without that count.
2. Target 2 ("one sentence to start") may be too aggressive — some ideas genuinely need clarifying questions. The target conflates "zero friction to start the tool" with "zero friction to start iterating."
3. The success criteria are self-assessed. Target 3 tries to address this by requiring documented evidence, but Targets 1 and 2 are still evaluated by the same agent that proposed them.
