## Move rationale
Failure modes (60) is the bottleneck. The v003 red team flagged compounding errors from deferred validation — the agent could build multiple iterations on a bad injection. Researched how autonomous systems handle this to ground the mitigation rather than inventing one from intuition.

## Research findings

- **Error compounding is the primary risk in autonomous agent loops.** State errors are silent and cumulative — every subsequent step builds on a faulty premise. "By the final stage, you have a confidently-written document that is subtly or substantially wrong." (Source: [Self-Correcting Multi-Agent AI Systems](https://medium.com/@sohamghosh_23912/self-correcting-multi-agent-ai-systems-building-pipelines-that-fix-themselves-010786bae2db))
- **Checkpoint-and-rollback is the standard mitigation.** Checkpoint at major milestones; if quality drops, roll back to the last known-good state rather than propagating the error forward. (Source: [Error Recovery in AI Agent Development](https://www.gocodeo.com/post/error-recovery-and-fallback-strategies-in-ai-agent-development))
- **Bounded improvement spaces** constrain what the agent can change, preventing runaway drift. (Source: [Self-Improving Data Agents](https://powerdrill.ai/blog/self-improving-data-agents))
- **Rollback triggers fire on quality score drops.** If scores drop by more than a defined threshold, automatic rollback occurs. (Source: [Partnership on AI](https://partnershiponai.org/wp-content/uploads/2025/09/agents-real-time-failure-detection.pdf))

## How this grounds the tool

The plan generator already has two of these mechanisms without naming them:
1. **Checkpoint**: Every variant is saved to `variants/vNNN.md`. The `current-best.md` is only updated when a variant wins on the bottleneck comparison. Failed variants are preserved but don't propagate.
2. **Bounded improvement space**: Each iteration applies exactly one move. The agent can't rewrite the entire idea in one step.

What's **missing** is an explicit rollback trigger. Currently, if a bad injection from an EC (Move 5) becomes current-best, subsequent iterations build on it with no mechanism to detect that the injection was wrong. The bottleneck rule only catches variants that are *locally* worse — it doesn't catch a globally bad direction that still looks better than the previous variant.

## New mechanism: drift detection

After every 3 iterations, the agent re-reads the seed and asks: **"Is the current-best still recognizably pursuing the same idea as the seed? If not, what changed and was the change deliberate?"**

This is a lightweight coherence check, not a full re-evaluation. It catches two failure modes:
1. **Injection drift**: An EC injection reframed the idea so much that it's now solving a different problem.
2. **Scope creep**: Successive iterations added capabilities until the idea is unrecognizably complex.

If drift is detected, the agent flags it in the iteration log and considers whether to roll back to the last pre-drift variant or continue with the new direction (documenting the choice).

## Idea

The cssi-plan-generator is an auto-improving business idea refinement tool.

### Design principle: front-load human input, then run autonomously

1. **Seed phase (interactive):** Agent drafts proposed answers from web research. User corrects/approves. User confirms the seed.
2. **Iteration phase (autonomous):** Full iteration budget, no mid-run confirmations. All decisions documented for post-run review.
3. **Plateau handling (autonomous):** Auto-stop after 3 consecutive non-improvements post-plateau-detection.
4. **Report phase (human review):** Flags injections, checkpoints, and scoring corrections.

### Error containment (grounded in research)

The methodology already has two containment mechanisms:
- **Variant checkpoints**: Every variant saved; current-best only updated on bottleneck wins. Failed variants don't propagate.
- **Bounded moves**: One move per iteration prevents runaway rewrites.

Added mechanism:
- **Drift detection (every 3 iterations)**: Agent re-reads the seed and checks whether current-best is still pursuing the same idea. Flags injection drift or scope creep. Considers rollback to last pre-drift variant.

## Red team (pre-scoring criticisms)

1. Drift detection every 3 iterations is a check, not a prevention. By the time drift is detected, 3 iterations may have been wasted. But the alternative (checking every iteration) adds overhead that slows the loop.
2. The "re-read the seed" check is subjective — the agent decides whether the idea has drifted, which is the same self-evaluation problem from v001. The structural mitigation is that the seed is a fixed document, so there's at least an objective anchor.
3. The research grounded the *problem* (compounding errors) but the *solution* (drift detection) is still invented. No source described this exact mechanism for document-level iteration. It's plausible but unvalidated.
