# Variant v001

## Move applied
Move 5: Identify and resolve a core tension using TOC Evaporating Cloud

## Move rationale
Failure modes is the bottleneck (45) because the tool scores its own output with no external check, creating a self-congratulation risk. This is a buried dilemma — the tool needs honest evaluation but is also the evaluator.

## Evaporating Cloud

**Objective (A)**: The iteration loop produces genuinely better variants over time

**Requirement B**: Scoring must be honest and calibrated
**Prerequisite D**: An external evaluator checks the scores

**Requirement C**: The loop must run autonomously without human intervention
**Prerequisite D'**: The agent scores its own output

**Conflict**: D (external evaluator) conflicts with D' (agent scores itself) — you can't have both full autonomy and external scoring.

### Assumptions

- **(1) A → B**: Honest scoring is necessary because inflated scores hide real weaknesses and send iterations in the wrong direction.
- **(2) A → C**: Autonomy is necessary because the whole point is to run N iterations without human babysitting.
- **(3) B → D**: An external evaluator is needed because self-evaluation is inherently biased.
- **(4) C → D'**: The agent must score because no one else is available during autonomous iteration.
- **(5) D ↔ D'**: You can't have both an external evaluator and pure self-scoring.

### Challenged assumption

**(3) B → D**: "An external evaluator is needed for honest scoring." This is wrong. What's actually needed is *structural resistance to inflation* — mechanisms that make it hard for the agent to drift upward. An external evaluator is one way to achieve this, but not the only way.

### Injection

Instead of requiring an external evaluator, build multiple anti-inflation mechanisms directly into the scoring process:

1. **Red team step**: Before scoring any variant, the agent must write 2-3 specific criticisms of the variant — things that are weaker, missing, or could fail. These criticisms are recorded in the variant file before scores are assigned.
2. **Anchored justifications**: Every score must reference a concrete element of the variant (a named mechanism, a specific claim, a particular gap). No score can be justified by a general impression.
3. **Regression check**: After scoring, the agent compares each dimension to the previous variant and must explain any score that went up by more than 10 points in a single iteration. Large jumps require explicit justification of what specifically changed to warrant them.
4. **Bottleneck lock**: The existing rule (lowest score determines the winner) already resists inflation — inflating one dimension doesn't help if the bottleneck is elsewhere. This is already in the rubric but its anti-inflation function should be made explicit.

These mechanisms don't eliminate bias, but they make lazy inflation structurally difficult. The agent can still run autonomously.

## Idea

The cssi-plan-generator is an auto-improving business idea refinement tool that applies 11 improvement moves drawn from TOC and TRIZ in an autonomous iteration loop.

**Self-evaluation integrity** is maintained through four structural mechanisms rather than requiring an external evaluator:

- **Red team step**: Before scoring, the agent writes 2-3 specific criticisms of the current variant. These are recorded in the variant file and visible to the human reviewing the run.
- **Anchored justifications**: Every score references a concrete element — a named mechanism, specific claim, or particular gap. No general-impression scores.
- **Regression check**: Any single-dimension score increase of >10 points in one iteration requires explicit justification of what changed.
- **Bottleneck comparison rule**: The lowest score determines the winner, so inflating non-bottleneck dimensions provides no advantage.

These mechanisms supplement the existing devil's advocate requirement for 85+ scores and the rubric-locking rule. Together they create a scoring environment where honest assessment is the path of least resistance.

**Three improvement goals for this run:**

1. **Reduce human dependency** — identify every place where the methodology asks a human to do something the agent could do, and redesign those steps.
2. **Sharpen for multiple audiences** — ensure the reference docs, output format, and interaction style work for a solo practitioner, a consultant with clients, and a stranger reading the repo.
3. **Stress-test coherence** — find internal contradictions, missing instructions, or ambiguous heuristics by actually following them under pressure.

## Red team (pre-scoring criticisms)

1. The four anti-inflation mechanisms are untested — they sound rigorous but could become pro-forma box-checking that the agent performs without genuine self-critique.
2. The regression check (>10 point jump) is arbitrary. Why 10? A 9-point jump could be just as suspect.
3. None of these mechanisms address the deeper problem: the agent may genuinely not know when it's wrong. Structural checks reduce lazy inflation but don't fix blind spots.
